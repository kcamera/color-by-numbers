# color-by-numbers

A "color by numbers" app that actually works well, for iPadOS.

Calm, kid-first, and honest: no ads, no in-app purchases, no reward loops,
no network — ever. Any picture your family loves becomes art your child
makes herself: source image → color-by-numbers template → finished art.

## Layout

| Path | What it is |
|---|---|
| `Sources/CBNKit/` | Pure Swift core: document model, import pipeline (M1), renderers. Builds on macOS and iOS |
| `Sources/cbnc/` | macOS CLI over CBNKit: import, preview, and the preset tuning workflow (M1) |
| `Tests/CBNKitTests/` | `swift test` from the repo root |
| `Samples/` | Hand-authored and starter templates |
| `App/` | The iPad app (SwiftUI). `project.yml` is the source of truth |
| `docs/` | Plan, design principles, and future collateral |

## Building

Requires Xcode 16+ and [XcodeGen](https://github.com/yonaskolb/XcodeGen)
(`brew install xcodegen`).

```sh
# Core library + CLI (from repo root)
swift test
swift run cbnc validate Samples/LittleSailboat/template.json

# Import an image and preview the result
swift run cbnc import picture.png --preset just-right
swift run cbnc render picture.json --mode outline

# iPad app
cd App && xcodegen        # regenerate the .xcodeproj (it is gitignored)
open ColorByNumbers.xcodeproj
```

`cbnc --help` lists every verb with a one-line summary of the desk workflow
they fit into; `cbnc <verb> --help` documents that verb's own flags in full
(`tune --help` in particular explains the `c<n>-d<n>-m<n>` filename
convention on its output images).

## The preset tuning loop

Presets live in `Sources/CBNKit/Resources/presets.json` — data, not code.
The app and CLI read the same file, so retuning never touches app code.
When an image imports badly, the loop is:

1. **Add the problem image** to `TestArt/` (it joins the corpus permanently
   — future tuning sessions inherit it).
2. **Sweep**: `swift run -c release cbnc tune TestArt -o /tmp/sheet
   --colors 6,10,16 --detail 0.35,0.6,0.85` (any comma-separated ranges;
   add `--min-region-mm` values to sweep the smallest-colorable-region
   floor too — a dot diameter in millimeters at display size, e.g.
   `--min-region-mm 12,8,4`). The `-c release` matters: `swift run`
   defaults to an unoptimized debug build, which runs the pipeline ~30×
   slower — minutes instead of seconds for a sweep.
3. **Look**: open `/tmp/sheet/index.html`. Each cell shows the composite
   render (boundary quality) and the outline render (what the child sees),
   labeled with its exact parameters and region/color counts.
4. **Bless**: copy the winning numbers into `presets.json`, run
   `swift test` (the corpus tests catch regressions), commit.

`TestArt/` is generated by `swift run cbnc make-testart` (synthetic,
deterministic, license-free); real public-domain flat art can be added
alongside the generated files at any time.

`cbnc` commands scan a directory's direct contents only, not
subdirectories — which `TestArt/local/` relies on to stay out of default
sweeps. It's also **gitignored**: licensed characters, watermarked
samples, real photos/3D renders (out of v1's flat-art scope), or anything
else you want to explore locally but never have redistributed via the
repo belongs there, organized into subdirectories however you like. See
`TestArt/local/README.md`. Point a command at it explicitly
(`cbnc tune TestArt/local`) when you want it.
